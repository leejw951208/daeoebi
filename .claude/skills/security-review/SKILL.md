# 보안 검토 (비밀번호 보관함)

이 제품은 1인용 암호화 비밀번호 보관함이다. 보안이 곧 제품이다. **적대적 관점**으로 "이 코드를 어떻게 깨뜨릴 수 있는가"를 따진다. 존재 확인이 아니라 악용 경로 추적이다.

## 검토 절차
1. 변경 파일 목록을 받아 보안 표면(암호화·인증·세션·CSRF·시크릿)에 닿는 부분을 추린다.
2. 아래 체크리스트를 영역별로 적용한다.
3. 발견을 심각도와 악용 시나리오와 함께 보고한다. 직접 고치기보다 수정 방향을 제시한다 (단순·명백한 경우만 직접 수정).

## 체크리스트

### 시크릿 누출
- 평문 비밀번호·암호화 키·복구코드·verifyToken·마스터가 **로그·응답 바디·에러 메시지·URL 쿼리·클라이언트 저장소(localStorage 등)**로 새는가.
- 복호화된 본문(`reveal`)이 인가되지 않은 상태에서 반환되는가. 마스터/잠금 가드가 빠진 경로는 없는가.
- 에러 스택·디버그 출력에 민감 데이터가 섞이는가.

### 암호화 (vault-crypto, argon2)
- AEAD(iv·ciphertext·authTag)가 올바르게 검증되는가. authTag 미검증·무시 경로가 있는가.
- iv(nonce)가 재사용되는가. 매 암호화마다 새로 생성되어야 한다.
- KDF 파라미터(argon2id memory/iterations)가 약화되지 않았는가. `kdfVersion` 호환 처리가 안전한가.
- 키가 메모리에서 idle 타임아웃 후 실제로 폐기되는가 (재잠금).

### 인증·세션·가드
- 모든 보호 엔드포인트에 가드(`PinGuard`/`vault-lock.guard`/`auth.guard`)가 걸렸는가. 신규 라우트가 보호망에서 누락됐는가.
- 세션 쿠키가 httpOnly·sameSite·secure인가 (`*-cookies.ts`). 세션 고정·만료 처리가 있는가.
- 잠금해제 실패 시 백오프/rate-limit(`vault-backoff.service.ts`, `PinCredential.failCount/lockUntil`)가 동작하는가. 우회 가능한가.

### CSRF
- 모든 상태 변경(POST/PATCH/DELETE) 요청이 CSRF 미들웨어(`*-csrf.middleware.ts`)로 보호되는가. 신규 쓰기 라우트가 누락됐는가.
- 클라이언트가 `X-Vault-Request` 등 요구 헤더를 부착하는가, 서버가 이를 검증하는가.

### 입력 검증
- DTO `class-validator`가 모든 외부 입력에 적용됐는가. import 컨테이너·파일 업로드 등 비정형 입력의 크기·형식 제한이 있는가.

## 보고 형식
영역별로 다음을 `_workspace/`에 기록하고 리더·엔지니어에게 전달한다:
- **심각도**: critical / high / medium / low
- **위치**: `파일:라인`
- **악용 시나리오**: 한 줄
- **수정 방향**: 한 줄

critical/high는 즉시 해당 엔지니어에게 통보한다. 확신이 없으면 "확인 필요"로 남기고 삭제하지 않는다. 추측성 지적은 보고하지 않는다.
