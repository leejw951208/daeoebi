# Writing QA Scenarios (daeoebi)

## Overview

대외비(daeoebi) 서비스의 **전체 기능 QA 시나리오 문서**를 만든다. 하나의 마스터 문서에 분류(feature area)별 섹션으로 테스트 케이스를 표로 정리한다. 절차는 **Playwright로 그대로 실행**할 수 있도록 구체적인 조작·셀렉터 수준으로 쓴다.

**핵심 원칙:** 이 스킬은 **시나리오를 작성**한다(실제 결과·Pass/Fail은 비워 둔다). 실행은 Playwright QA 세션에서 채운다. 절차가 모호하면 QA가 불가능하므로, 각 절차는 "무엇을 눌러 무엇을 확인" 수준까지 구체화한다.

## When to Use

- 전체 기능에 대한 QA 시나리오 문서를 새로 만들거나 갱신할 때
- 새 기능을 추가한 뒤 해당 분류에 시나리오를 덧붙일 때
- Playwright QA를 돌리기 전에 실행 대상(케이스 목록)을 정의할 때

## 산출물

- 위치: `docs/qa/qa-scenarios.md` (단일 마스터 문서)
- 구성: 상단 요약 + 분류별 섹션, 각 섹션에 아래 7컬럼 표

| 컬럼 | 내용 |
|------|------|
| 분류 | 기능 영역(섹션과 동일). 예: 인증, 가계부-지출 |
| 시나리오 ID | `<AREA>-NNN` (예: `AUTH-001`, `EXPENSE-003`). 3자리 zero-pad |
| 사전 조건 | 실행 전 상태(로그인/잠금해제 여부, 시드 데이터, 특정 화면) |
| 절차 | 번호 매긴 조작 단계. Playwright 액션으로 옮길 수 있게 구체적으로 |
| 예상 결과 | 각 절차 후 관찰 가능한 결과(문구·상태·이동) |
| 실제 결과 | **비워 둔다** (QA 실행 시 기입) |
| Pass/Fail | **비워 둔다** (QA 실행 시 기입) |

## 분류 커버리지 체크리스트 (전 기능)

각 항목마다 정상 흐름 + 주요 예외/경계 케이스를 만든다.

- **인증(AUTH)** — 온보딩(패스키 등록), 잠금해제(패스키/PRF), 복구코드 잠금해제, 복구코드 5회 실패 → 60초 잠금, 세션 만료/재잠금
- **사이트(SITE)** — 목록·검색, 생성·수정·삭제, 기본 사이트(가장 오래된 것) 노출
- **비밀번호 카테고리(CAT)** — 사이트 내 카테고리 CRUD
- **시크릿(SECRET)** — 항목 생성(필드 추가/민감필드 마스킹)·상세·수정·삭제, 값 복사 후 클립보드 자동 클리어
- **백업(BACKUP)** — export 내려받기, import reject/skip/replace 3모드, 무결성 오류(참조 누락) 거부
- **가계부-예산(BUDGET)** — 월 예산 설정·수정(금액 단건)
- **가계부-지출(EXPENSE)** — 지출 추가(금액·항목·카테고리·날짜)·수정·삭제, 대시보드 반영
- **가계부-고정지출(RECURRING)** — 고정 지출 템플릿 CRUD, 월 자동 생성(멱등)
- **가계부-카테고리(ASSETCAT)** — 카테고리 CRUD, HEX 색 입력·검증, 이름 중복 방지(409), 삭제 시 지출 미분류 처리
- **가계부-대시보드(DASH)** — 남은 예산·예산/지출·카테고리별 분해·달력·일자 상세, 카테고리별 색·집계 정확성
- **데모(DEMO)** — `/demo` 인증 없이 접근, 비밀번호↔가계부 탭, 데모 상호작용(변경 미저장)
- **PWA/설치(INSTALL)** — 설치 배너 노출 조건

> 실제 라우트·컴포넌트는 `apps/web/app/(vault)/`, API는 `apps/api/src/`를 참고해 최신 동작으로 맞춘다. 제품 정의는 `docs/PRD.md`.

## 절차 작성 규칙 (Playwright 실행 가능하게)

- 조작은 **역할·라벨 기반 셀렉터**로 옮길 수 있게 쓴다: 버튼 이름("+ 추가"), `getByLabel`("금액"), `getByRole("dialog", { name: "카테고리 관리" })`.
- 화면 진입은 URL 또는 탭/버튼 클릭으로 명시. 웹 `http://localhost:3010`, API `:4010`.
- **인증 우회:** 로컬/테스트는 dev-auth 우회(`apps/web/lib/dev-auth.ts`의 `devUnlock`)를 쓴다 — 기존 `apps/web/tests/e2e/*.spec.ts` 패턴을 따른다(`playwright.e2e.config.ts`). 시나리오 사전 조건에 "dev-unlock 상태"로 적으면 된다.
- 고유 데이터가 필요하면 절차에 "고유 이름(예: 접미어)"을 명시해 반복 실행 충돌을 피한다.
- 예상 결과는 **관찰 가능**해야 한다: 특정 문구 노출, 목록에 항목 추가, 버튼 `aria-busy`, 다이얼로그 닫힘 등.

## 시나리오 예시 (모델 — 실제 문서는 실제/Pass·Fail 비움)

| 분류 | 시나리오 ID | 사전 조건 | 절차 | 예상 결과 | 실제 결과 | Pass/Fail |
|------|-----------|---------|------|---------|---------|-----------|
| 가계부-카테고리 | ASSETCAT-002 | dev-unlock 상태, 자산 화면 | 1. "카테고리 관리" 클릭 2. 이름에 고유값 입력 3. 색 HEX에 `#3bb273` 입력 4. "+ 추가" 클릭 | 목록에 새 카테고리가 색 점과 함께 노출, 에러박스 없음 |  |  |
| 가계부-카테고리 | ASSETCAT-003 | dev-unlock 상태, "식비" 존재 | 1. "카테고리 관리" 클릭 2. 이름에 `식비` 입력 3. "+ 추가" 클릭 | "같은 이름의 카테고리가 이미 있습니다." 에러 노출, 추가 안 됨 |  |  |

## 작성 절차

1. 커버리지 체크리스트의 각 분류에 대해 실제 코드/화면을 확인해 현재 동작을 파악한다(구식 시나리오 금지). **먼저** 해당 화면 컴포넌트에서 버튼·라벨·placeholder·aria-label 의 한국어 원문을 수집한다(예: `apps/web/app/(vault)/**` grep). 이 원문이 없으면 절차를 Playwright로 옮길 수 없다. 확인 못 한 정확 문구는 절차에 `(문구 확인 필요)`로 표시해 실행 시 확정한다.
2. 분류별로 정상 흐름 1+ 개, 주요 예외/경계 1+ 개 케이스를 만든다.
3. 시나리오 ID는 분류 접두어 + 3자리 순번으로 유일하게.
4. 절차는 Playwright 액션으로 옮길 수 있는 구체 수준으로.
5. 실제 결과·Pass/Fail 컬럼은 **빈칸**으로 남긴다.
6. `docs/qa/qa-scenarios.md`에 저장. 상단에 문서 목적·실행 방법(dev-unlock, `playwright.e2e.config.ts`)·갱신일 요약을 둔다.

## Common Mistakes

- **절차가 추상적** ("카테고리를 추가한다") → Playwright로 못 옮김. "이름에 X 입력 → '+ 추가' 클릭"처럼 조작 단위로.
- **예상 결과가 비관찰적** ("정상 동작") → "목록에 항목이 노출"처럼 확인 가능한 것으로.
- **실제 결과/Pass·Fail을 미리 채움** → 이 스킬은 작성만. 실행 전까지 비운다.
- **커버리지 누락** → 체크리스트의 모든 분류를 훑고, 최소 정상+예외 케이스를 확보.
- **구식 동작 반영** → 코드/PRD로 현재 동작을 확인 후 작성.
