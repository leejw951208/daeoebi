# Spec: 정기 지출 관리

> 카드값, 관리비, 생활비 등 매월/매주/매년 반복되는 지출을 한 곳에서 등록·조회·결제 처리하는 로컬 전용 1인 웹 서비스의 첫 기능.

## 배경

여러 카드 청구액, 통신비, 구독료, 월세/관리비가 서로 다른 날짜에 빠져나가며, 자동이체 누락이나 카드 한도 초과를 사후에 인지하는 일이 잦다. 가계부 앱은 다인 사용·클라우드 전제라 과하고, 스프레드시트는 "다음 결제일"을 자동 계산해 주지 못한다. 이 기능은 본인 한 명이 로컬에서만 실행해 정기 지출을 등록하고, 향후 N개월의 결제 예정 인스턴스를 자동 생성해 캘린더와 리스트로 확인하며, 실제 결제 여부와 금액 차이를 기록할 수 있게 한다. 후속 기능인 비밀번호 관리와 같은 DB·앱 셸을 공유하도록 데이터 모델과 인증 경계를 설계한다.

## 기능 목록

### 1. 정기 지출 마스터 등록·수정·삭제

정기적으로 발생하는 지출의 "원본"을 관리한다.

**동작 방식**
- 사용자가 "새 정기 지출" 버튼으로 이름, 카테고리, 예상 금액, 통화(기본 KRW), 반복 주기(매월/매주/매년), 결제일 지정 규칙(매월 N일, 매주 요일, 매년 M월 N일), 시작일, 종료일(선택), 결제 수단, 메모를 입력한다.
- 저장 시 시작일 ~ 시작일+12개월 범위의 결제 예정 인스턴스가 자동 생성된다.
- 마스터를 수정하면 "오늘 이후의 미발생 인스턴스"만 재생성되고 과거(완료/스킵)는 그대로 보존된다.
- 마스터를 삭제하면 마스터는 soft delete(`isActive=false`)되고, 미래 미발생 인스턴스는 제거되며 과거 이력은 남는다.

**포함 범위**
- 매월/매주/매년 반복.
- 결제일이 말일을 초과하는 달의 처리(예. 매월 31일 → 2월은 28/29일로 조정).
- KRW 표시(₩123,456) 기본, 입력 시 자동 천 단위 구분.

**제외 범위**
- 격주, "매월 마지막 영업일" 같은 복잡한 규칙.
- 다중 통화 환산. 입력 통화만 표시한다.
- 외부 은행/카드 명세 자동 동기화.

### 2. 결제 예정 인스턴스 조회·완료 처리

생성된 결제 예정 인스턴스를 캘린더와 리스트로 확인하고, 실제 결제 결과를 기록한다.

**동작 방식**
- 대시보드는 "이번 달 예상 합계", "다음 7일 예정", "이번 달 미완료 항목 수"를 카드로 보여준다.
- 캘린더 뷰는 월 단위로 결제 예정 인스턴스를 점/금액 칩으로 표시한다.
- 리스트 뷰는 다음 30일(기본) 또는 임의 범위의 인스턴스를 카테고리/카드/상태로 필터링해 보여준다.
- 인스턴스를 클릭하면 "완료(실제 금액 입력)", "스킵(사유 메모)", "예정으로 되돌리기"를 선택할 수 있다.
- 예상 금액과 실제 금액이 다르면 차이가 강조되어 표시된다.

**포함 범위**
- 카드값처럼 "예상 vs 실제"가 매월 달라지는 경우의 기록.
- 인스턴스별 메모.

**제외 범위**
- 인스턴스를 통째로 추가/삭제(반드시 마스터를 통해서만 변경).
- 영수증 이미지 첨부.

### 3. 카테고리·기간별 합계 및 CSV 내보내기

지출 패턴을 빠르게 점검하기 위한 보조 기능.

**동작 방식**
- 임의 기간(이번 달/지난 달/올해/사용자 지정)에 대해 카테고리별 합계, 결제 수단별 합계, 상태별 합계를 표로 본다.
- 같은 화면에서 "CSV로 내보내기"를 누르면 인스턴스 목록(날짜, 이름, 카테고리, 예상, 실제, 상태, 결제 수단)을 UTF-8 BOM 포함 CSV로 다운로드한다.

**포함 범위**
- 합계는 실제 금액이 입력된 항목은 실제 금액, 미입력은 예상 금액 기준으로 계산하고 "기준" 컬럼을 표시한다.
- CSV는 엑셀 한글 호환을 위해 UTF-8 BOM 포함.

**제외 범위**
- 그래프/차트.
- 예산 설정 및 알림.

## 입출력 / 제약 조건 / 예외 케이스

| 항목 | 내용 |
|------|------|
| 입력 | 정기 지출 마스터 폼(이름, 카테고리, 금액, 통화, 반복 규칙, 시작/종료일, 결제 수단, 메모), 인스턴스 완료/스킵 입력(실제 금액, 메모), 조회 필터(기간, 카테고리, 상태). |
| 출력 | 대시보드 요약 카드, 캘린더(월), 인스턴스 리스트, 카테고리 합계 표, CSV 파일. |
| 제약 조건 | 단일 사용자, 프론트엔드와 백엔드 모두 localhost(127.0.0.1) 바인딩, SQLite 단일 파일 DB, 외부 네트워크 호출 없음. 금액은 정수 최소 단위로 저장(원/센트). 날짜는 사용자 로컬 타임존(Asia/Seoul 기본) 기준 자정. 인증은 이번 단계 미적용. 추후 비밀번호 기능 추가 시점에 NestJS Guard 자리에 도입. |
| 예외 케이스 | 매월 31일 결제 → 30/29/28일로 자동 조정. 시작일이 결제일 이후 → 다음 주기부터 생성. 종료일이 시작일 이전 → 검증 에러. 마스터 삭제 시 과거 인스턴스 보존. 동일 인스턴스를 두 번 "완료" 처리 시 마지막 입력이 덮어쓴다. DB 파일 권한 부족 시 시작 단계에서 명시적 에러 출력. |

## 아키텍처 개요

이 기능은 모노레포로 구성된 두 개의 독립 프로세스로 동작한다.

- **`apps/web`**. Next.js(App Router, TypeScript) 클라이언트. UI 렌더링과 사용자 입력 폼, 캘린더/리스트/대시보드를 담당한다. 데이터는 직접 DB를 만지지 않고 `apps/api`의 HTTP 엔드포인트를 호출해 가져온다.
- **`apps/api`**. NestJS(TypeScript) 백엔드. Prisma 스키마와 마이그레이션, 비즈니스 로직(반복 규칙 계산, 마스터/occurrence 동기화), HTTP REST 엔드포인트를 모두 보유한다.
- **DB**. SQLite 단일 파일(`apps/api/data/secrets-manager.db`). `apps/api`만 직접 접근한다.
- **네트워킹**. 두 프로세스 모두 `127.0.0.1`에만 바인딩한다(web 3000, api 4000). 외부 네트워크 노출 없음.

프론트엔드는 `axios` 인스턴스 한 개에 `NEXT_PUBLIC_API_BASE_URL`(기본 `http://127.0.0.1:4000`) 환경변수를 주입해 API를 호출한다.

## 채택 근거 및 기각된 대안

| 결정 | 채택 근거 | 기각된 대안 |
|------|----------|-----------|
| Next.js(`apps/web`) + NestJS(`apps/api`) 모노레포 분리 | 후속으로 비밀번호 관리, 모바일 클라이언트, CLI 등 다른 프런트를 붙일 가능성을 열어 둔다. NestJS의 모듈/Provider/Guard 구조가 도메인 분리·인증 도입에 자연스럽다. 백엔드를 독립 프로세스로 두면 Prisma·반복 규칙 로직을 UI 렌더링 사이클과 분리해 테스트하기 쉽다. | 풀스택 Next.js 단일 앱(API Route + Server Action으로 처리). 1인 사용에는 충분하지만, 도메인 로직이 RSC/Route Handler에 흩어져 후속 비밀번호 도메인이 추가될 때 경계가 모호해진다. 또한 Next.js 빌드/배포 사이클에 백엔드 변경이 묶여 반복 규칙 단위 테스트와 마이그레이션 작업이 무거워진다. |
| pnpm workspace 기반 모노레포(`apps/*`, `packages/*`) | pnpm은 디스크 사용량과 설치 속도가 빠르고 워크스페이스 의존성 해석이 엄격해, 두 앱이 공유 타입(예. DTO)을 안전하게 import할 수 있다. Next.js, NestJS 모두 공식 가이드에서 1급 지원. | npm workspace(엄격성 부족, hoisting 이슈). Nx/Turborepo(이번 단계 규모엔 도구 부담이 가치 대비 큼. 필요 시 후속 단계에서 도입 검토). |
| SQLite 파일(`apps/api/data/secrets-manager.db`) | 외부 DB 서버 불필요, 백업은 파일 복사, 로컬 1인에게 충분. NestJS + Prisma의 SQLite 드라이버는 안정적. | Postgres(운영 부담 큼). IndexedDB(클라이언트 측 저장이라 NestJS 백엔드와 어긋남). |
| Prisma 스키마는 `apps/api`에서 관리 | DB에 접근하는 유일한 프로세스가 `apps/api`이므로 스키마·마이그레이션·생성된 클라이언트를 한 곳에 둔다. 프런트는 DB를 알 필요가 없다. | `packages/db`로 분리(이번 단계엔 공유할 다른 소비자가 없어 과잉). |
| 마스터 + Occurrence 분리 모델 | 카드값처럼 "예상 vs 실제"가 매월 다른 케이스를 기록 가능. 과거 이력 보존이 자연스러움. | 마스터만 두고 매번 계산(예상 vs 실제 기록 불가). |
| 시작일+12개월치 occurrence 사전 생성 | 캘린더/대시보드 조회 시 단순 SELECT로 처리. 마스터 수정 시 미래만 재생성. | 매 조회 시 즉석 계산(로직 복잡, 인스턴스 메모/실제금액 보존 불가). |
| 매월 31일 등 말일 초과 → 해당 월 말일로 자동 조정 | 사용자 의도(매월 결제) 보존, 누락 방지. | 다음 달로 미루기(같은 달 결제를 건너뛰어 합계 왜곡). |
| 인증 이번 단계 미적용, NestJS Guard 자리만 마련 | 로컬 전용·1인 사용 전제. 비밀번호 기능과 함께 도입하는 편이 자연스럽다. NestJS의 `CanActivate` Guard 인터페이스에 빈 통과 구현을 두어 후속 단계에 교체만 하면 된다. | 이번 단계부터 마스터 패스워드 도입(범위 초과). 인증 자리를 비워두지 않음(후속 도입 시 라우트 전반 수정 필요). |
| 한국어 UI, KRW 기본 표시 | 본인 사용 케이스에 정확히 맞춤. | 영문/다국어(과잉). |
| 외부 접속 차단(`127.0.0.1` 바인딩, web/api 모두) | 1인 로컬 사용 전제, 공유 와이파이에서의 우발적 노출 방지. 백엔드가 분리되었으므로 api 포트도 명시적으로 잠근다. | 0.0.0.0 바인딩(불필요한 노출 위험). |

## 비기능 요건

| 항목 | 기대치 |
|------|--------|
| 성능 | 로컬 SQLite 기준 Next.js 페이지 초기 렌더 200ms 이내(api round-trip 포함). occurrence 1만 건까지 캘린더 월 뷰가 100ms 이내 응답. |
| 보안 | web/api 모두 127.0.0.1 바인딩, 외부 네트워크 호출 없음, DB 파일은 OS 사용자 권한(0600 권장)으로 저장, gitignore. NestJS에 `AuthGuard`(이번 단계엔 항상 통과) 자리만 마련해 두고, 후속 단계에서 단일 마스터 패스워드를 같은 Guard 안에 구현한다. CORS는 `http://127.0.0.1:3000`만 허용한다. |
| 확장성 | 후속으로 비밀번호 관리 도메인이 `apps/api`에 `VaultModule`로 추가되고, `apps/web`에는 `/vault` 라우트가 추가되는 것을 가정한다. NestJS는 도메인 모듈(`ExpensesModule`, 향후 `VaultModule`)로 경계를 두고, Next.js는 `app/(expenses)`, 향후 `app/(vault)` 라우트 그룹으로 분리한다. 공유 타입(DTO)이 늘어나면 `packages/shared`를 도입할 수 있도록 워크스페이스 구조를 유지한다. |

## 용어 정의

| 용어 | 설명 |
|------|------|
| 정기 지출 마스터(RecurringExpense) | 반복 규칙을 포함한 지출의 원본 정의. 사용자가 등록·수정·삭제하는 단위. |
| 결제 예정 인스턴스(ExpenseOccurrence) | 마스터로부터 특정 날짜에 생성된 한 번의 결제 항목. 상태(SCHEDULED/PAID/SKIPPED)와 실제 금액을 기록한다. |
| 예상 금액 | 마스터에 입력된 금액. 인스턴스 생성 시 복사된다. |
| 실제 금액 | 인스턴스 완료 처리 시 사용자가 입력한 실제 결제 금액. 카드값처럼 변동되는 항목용. |
| 결제 수단 | 카드명/계좌 별칭 등 사용자가 자유롭게 입력하는 라벨(이번 단계에선 별도 마스터 없이 문자열). |
